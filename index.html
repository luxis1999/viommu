IOMMU Introduction

IOMMU is I/O Memory Management Unit, which is used to do DMA address translation. Similar with what memory management unit does. IOMMU brings in I/O Virtual Address (IOVA) in DMA transactions. With IOMMU, system is well protected. Most hardware vendors have made IOMMU as a key feature on ther platform. E.g. Intel® Virtualization Technology (Intel® VT) for Directed I/O (Intel® VT-d). Per the latest Intel® VT-d 3.0 specification, it provides the following features:

- DMA remapping (DMA isolation)

- Interrupt remapping (interrupt isolation)

- Interrupt posting (efficient interrupt injection)

- Shared Virtual Memory (efficient worload submission)

- x2apic (> 255 CPU support)

- Scalable IOV (scalable & efficient I/O sharing)

Industry Trend

Scalable I/O Virtualization
Scalable IOV (S-IOV) a new hardware assisted I/O virtualization architecture published by Intel in June 2018. Kevin Tian (Principal Engineer from Intel) gave an introduction on it at the Linux Conference Beijing 2018 happened in the same month. With extensions in hardware and software, this architecture gives efficient and scalable I/O sharing to meet hyperscale I/O sharing requirement in cloud computing.

Device Assignment
Device assignment is a capability provided by IOMMU. It gives direct access of physical device to userspace applications (e.g. userspace device driver and device emulation (Qemu)). In virtualization environment, it gives bare-metal I/O peformance. In recent years, it is more and more popular to use device assignment in cloud.

- For one, emering accelerators (GPU, FPGA, NVMe, QAT) are hard to emulate with good perfomance

- For two, more and more I/O device offloading done by assignment in CSP

- For three, Scalable IOV (S-IOV) providescalable resource pool with lower cost than former SR-IOV

No doubt, with the techonlogy evolution, I/O assignment is not that costing as before. Then people may be more curious on the usage after assignment

vIOMMU

As mentioned above, industry is more and more care about the things after device assignment. e.g. is it secure to run userspace driver in guest or if it is able to apply SVM programming model.. Of course, answer is yes as long as VMM (Virtual Machine Monitor) exposes a Virtual IOMMU (vIOMMU) to guest.

With vIOMMU exposed to guest, guest can ideally benifit from all the aforementioned IOMMU features. Let's talk about DMA remapping and SVM. As these two features give guest secure and efficient device usage.

- guest IOVA (DMA isolation)

Latest Scalable Mode support since Linux kernel 5.0, Qemu v4.0.0-rc4

- guest SVM (efficient workload submission)

Try vIOMMU

Take Intel VT-d as an example, so far user can expose guest IOVA to VM. Guest SVM support is in progress.

Device assignment via VFIO

1) Load VFIO driver

$ sudo modprobe vfio

$ sudo modprobe vfio-pci

2) Bind device to vfio-pci driver

$ echo sgement_id:bus_id:devn_id.func_id > /sys/bus/pci/devices/sgement_id:bus_id:devn_id.func_id/driver/unbind

$ echo vendor_id device_id > /sys/bus/pci/drivers/vfio-pci/new_id

3) assign device to VM

"-device vfio-pci,host=bus_id:devn_id.func_id"

Device assignment with vIOMMU

Use the following command in Qemu command line (with strict order)

"intel-iommu,caching-mode=on,scalable-mode=on"

"-device vfio-pci,host=bus_id:devn_id.func_id"
